### Lost-update race (critical — must use grouped runner on PROD)
Initial QA mass at conc=3 across mixed classes silently lost 742 inserts. Two workers on same class:
- Worker A: GETDETAIL reads `[a,b,c]` → NEW = `[a,b,c,X]` → CHANGE (writes)
- Worker B (concurrent, same class, no SAP enqueue conflict): GETDETAIL reads `[a,b,c]` (BEFORE A's commit propagates to read buffer) → NEW = `[a,b,c,Y]` → CHANGE → **overwrites A's NEW, X lost**

Both BAPI calls return `Added via BAPI - CDHDR logged` success message, but only Y persists.

**Fix:** group pool by class, run classes parallel but pairs serial within class. See `v2_grouped_runner.py`.

Confirmed clean: 1954 / 1954 KSML + 1954 / 1954 CDHDR after grouped retry.

## 6. Bugs hit during build (lessons)

1. **Direct INSERT to KSML = no CDHDR** → auditor flag. Must use BAPI_CLASS_CHANGE. Document this.
2. **NCo proxy "Function parameter RETURN is unknown"** — proxy NCo SDK caches FM metadata. Renaming FM (e.g. V2 → BAPI suffix) gives fresh cache slot. RETURN must be TABLES BAPIRET2 in Z wrapper signature.
3. **BAPI_CLASS_GETDETAIL RETURN is IMPORTING (single), not TABLES** — if you put it under TABLES, NCo throws same "unknown" error. (Per memory `feedback_bapi_fixedasset_create1_return_export.md` — same pattern as FA worker.)
4. **DEV ATINN ≠ QA/PROD ATINN** — char internal IDs are env-local. Always pass `IV_ATNAM` (string) to V2 FM, never raw IMERK. CLINT is env-local too — pass `IV_CLASS` (class string) instead.
5. **Lost-update race on BAPI_CLASS_CHANGE concurrent same-class calls** — not protected by SAP enqueue when no other user holds lock. Driver MUST serialize per class.
6. **`BRONZE_BOT being processed` error** — SAP enqueue collision with concurrent BRONZE_BOT process (ARS/replenishment reads class). Retry serially with 2s backoff resolves.
